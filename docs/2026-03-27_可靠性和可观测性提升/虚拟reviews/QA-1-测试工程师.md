# QA-1 评审：测试与边界条件

> 评审对象：openNiuMa AI 自治研发编排器升级设计（合并版）
> 评审角色：资深测试工程师，聚焦可测试性、边界条件、竞态条件、错误处理完备性、测试策略充分性

---

## 🔴 严重（必须在实现前解决）

### S-1: state.py 文件锁缺少死锁恢复机制

**问题描述：** 设计提到 state.py 使用文件锁保护 loop-state.json 的读写，但没有定义锁超时、锁文件残留清理、以及进程异常退出后锁未释放的处理。在多 worker 并发场景下，如果持锁进程被 kill -9（例如 stall 检测触发的 kill），锁文件会残留，导致所有后续操作永久阻塞。

**影响分析：** 编排器完全停转。stall 检测 kill worker 后，如果该 worker 正在写 state，锁文件残留 → 其他 worker 和 reconcile 都无法获取锁 → 整个 loop 死锁。这是生产环境下的致命问题。

**建议修改：**
1. 文件锁必须带 TTL（建议 30 秒），超时自动释放并记录 warn 日志
2. 锁文件记录 PID，获取锁前检查 PID 是否存活（`os.kill(pid, 0)`）
3. 启动时清理陈旧锁文件（PID 不存在的锁）
4. 测试用例：模拟持锁进程被 kill，验证另一进程能在 TTL 后获取锁

---

### S-2: reconcile.py 的 kill 与 state 写入存在竞态窗口

**问题描述：** reconcile 检测到 stall 后执行 kill + 重置状态为 pending。但 kill 是异步的——被 kill 的 worker 进程可能在收到信号后、退出前的瞬间完成了任务并写入了 state（标记为 done）。reconcile 随后将其重置为 pending，导致已完成的任务被重复执行。

**影响分析：** 任务重复执行，可能产生重复的 git commit、重复的分支、重复的 PR。在有副作用的操作（如数据库迁移、部署）中后果严重。

**建议修改：**
1. kill 后等待进程确认退出（`waitpid` 或轮询 `/proc/{pid}`），再修改 state
2. state 写入使用版本号或时间戳乐观锁，reconcile 只在版本未变时才重置
3. 或者：kill 前先将 state 标记为 `killing`，worker 退出时检查此标记决定是否写入 done
4. 测试用例：模拟 worker 在被 kill 的同时完成任务，验证最终状态一致性

---

### S-3: 失败分类基于关键词匹配，存在严重误判风险

**问题描述：** failure.py 仅分析 session log 尾部 50 行的关键词来判定失败类型。这个方法有多个边界问题：
1. 日志尾部 50 行可能不包含真正的错误信息（例如错误发生在第 51 行之前，后面是清理日志）
2. 关键词可能出现在正常输出中（如 `"CONFLICT"` 出现在代码注释、测试用例名称中）
3. 多种失败类型同时出现时（如 network 导致 gate 失败），优先级规则未定义
4. `context window` 等关键词依赖 claude CLI 的具体输出格式，未来版本可能变化

**影响分析：** 误判导致错误的重试策略。例如把 gate 失败误判为 conflict，触发 merge 重试而非门禁重试；或者把 context 耗尽误判为 unknown，选择跳过而非清空续传。

**建议修改：**
1. 增加日志扫描行数到 200 行，或从尾部向上搜索到第一个错误标记
2. 关键词匹配使用上下文感知（如 `CONFLICT` 必须出现在 `git merge` 输出的上下文中）
3. 明确定义优先级：network > context > permission > conflict > gate > unknown
4. 为每种失败类型编写至少 5 个正例和 3 个反例（容易误判的场景）的测试用例
5. 增加 `confidence` 字段，低置信度时走 unknown 路径而非猜测

---

### S-4: 降级流程中"暂停 loop"缺少自动恢复路径

**问题描述：** 当有下游依赖的任务达到重试上限时，降级流程选择"暂停 loop"。但设计没有说明：
1. 暂停后如何恢复？需要人工干预还是有自动恢复条件？
2. 暂停期间其他无依赖关系的任务是否继续执行？
3. 暂停后 stall 检测、inbox 扫描是否还在运行？
4. 如果操作者长时间不在（无人值守场景），loop 可能停在这里数小时

**影响分析：** 无人值守场景下，一个有依赖的任务失败可能导致整个编排器停转，其他无关任务也被阻塞。

**建议修改：**
1. 暂停应该只阻塞有依赖关系的任务链，其他独立任务继续执行
2. 定义明确的恢复机制：`inbox/RESUME` 信号或超时自动恢复（配置项）
3. 暂停期间 reconcile 和 inbox 扫描必须继续运行（否则无法接收 RESUME 信号）
4. 测试用例：暂停后投递 RESUME 信号，验证恢复执行

## 🟡 中等（应在实现中解决）

### M-1: workflow.yaml 热重载缺少 schema 校验

**问题描述：** config.py 热重载时只检查"能否解析 YAML"，但没有 schema 校验。用户可能写出语法正确但语义错误的配置，例如 `stall_timeout_sec: "abc"`（字符串而非整数）、`max_concurrent: -1`（负数）、或缺少必填字段。

**影响分析：** 错误配置在运行时才暴露，且可能以难以理解的方式失败（如 Python 类型错误、比较失败等）。

**建议修改：**
1. 定义 workflow.yaml 的 JSON Schema 或 Python dataclass，加载时校验类型和范围
2. 数值字段增加范围校验（如 `stall_timeout_sec >= 60`、`max_concurrent >= 1`）
3. 必填字段缺失时使用默认值并记录 warn，而非静默通过
4. 测试用例：各种畸形 YAML（缺字段、类型错误、范围越界），验证报错信息清晰

---

### M-2: Prompt 模板严格模式未定义"部分渲染失败"的行为

**问题描述：** 设计说"未知变量报错退出"，但没有说明：一个模板中有 10 个变量，前 8 个正常，第 9 个未知——此时的行为是什么？已经渲染的部分是否回滚？报错信息是否指明哪个变量未知？

**影响分析：** 如果变量名拼写错误（如 `{{gate_comand}}`），编排器直接退出，但操作者可能很难定位是哪个 prompt 的哪个变量出了问题。

**建议修改：**
1. 渲染前先做一遍 dry-run 扫描所有 `{{var}}`，收集全部未知变量后一次性报错
2. 报错信息包含：文件名、行号、未知变量名、可用变量列表
3. 测试用例：模板含 1 个未知变量、含多个未知变量、变量名接近已有变量（建议拼写纠正）

---

### M-3: stats.json 并发写入无保护

**问题描述：** 多 worker 并发执行时，多个 worker 可能同时完成并调用 `stats.py record-session` 写入 stats.json。设计中 state.py 有文件锁，但 stats.json 的写入没有提到锁保护。

**影响分析：** 并发写入导致 JSON 损坏（部分写入、交叉写入），丢失统计数据。

**建议修改：**
1. stats.py 写入时复用 state.py 的文件锁机制（或独立锁文件）
2. 写入使用原子操作：先写临时文件，再 rename
3. 测试用例：多线程/多进程并发写入 stats.json，验证数据完整性

---

### M-4: 孤儿回收的"无活跃 worker"判定不够健壮

**问题描述：** reconcile.py 将 `in_progress` 但无活跃 worker 的任务重置为 `pending`。但"活跃 worker"的判定方式不明确——是检查 PID 存活？检查 workers/ 目录下的文件？如果 worker 刚启动还没来得及注册（写入 PID 文件），就会被误判为孤儿。

**影响分析：** 任务被错误回收为 pending，可能导致同一任务被两个 worker 同时执行（原 worker 还在，新 worker 又领取了）。

**建议修改：**
1. 明确 worker 注册时序：先注册 PID → 再标记任务 in_progress → 开始执行
2. 孤儿回收增加宽限期（grace period），如任务标记为 in_progress 后 60 秒内不回收
3. 判定"无活跃 worker"需同时满足：PID 不存在 + 日志无更新 + 超过宽限期
4. 测试用例：模拟 worker 启动慢（注册前延迟），验证不被误回收

---

### M-5: cancel 信号的交付保证不足

**问题描述：** 取消通过 `inbox/CANCEL-{id}` 文件实现，reconcile 轮询检测。存在以下问题：
1. inbox 扫描间隔 60 秒，取消最多延迟 60 秒才被发现
2. 如果任务正好在 CANCEL 文件创建后、reconcile 扫描前完成了，取消信号被忽略——但已完成的任务已经产生了副作用
3. 如果 CANCEL 文件在 reconcile 处理前被意外删除（如手动清理 inbox），取消丢失

**影响分析：** 取消不及时、取消无法撤回副作用、取消信号可能丢失。

**建议修改：**
1. 取消检测可以比 inbox 扫描更频繁（如 10 秒一次），或在每次 reconcile 中都检查
2. CANCEL 文件处理后移动到 `inbox/.processed/` 而非删除，保留审计记录
3. 明确语义：取消 = 尽力终止 + 标记 cancelled，不保证撤回已产生的副作用（文档中说明）
4. 测试用例：任务 in_progress 时取消、任务 pending 时取消、任务已完成时取消——验证三种情况下的行为

---

### M-6: init.sh 的 Layer 2（AI 生成）输出不可控

**问题描述：** init.sh 调用 `claude -p` 生成 `_common-rules.md`，但 AI 输出具有不确定性：
1. 输出格式可能不符合预期（不含 `{{gate_command}}`、包含多余解释文字）
2. 输出可能包含幻觉信息（描述项目中不存在的规范）
3. 同一项目多次运行得到不同结果，影响可复现性
4. claude CLI 的 `--output-format text` 行为可能在版本间变化

**影响分析：** 生成的 _common-rules.md 质量不稳定，可能注入错误规范导致 AI agent 行为异常。

**建议修改：**
1. AI 生成后做基本校验：必须包含 `## 硬性红线`、`## 硬性门禁`、`{{gate_command}}`
2. 校验失败则回退到模板 + warn 日志
3. 考虑对 AI 输出做后处理：strip 非 markdown 内容、确保 `{{gate_command}}` 存在
4. 测试用例：mock claude CLI 返回各种异常输出（空输出、纯文本、缺少关键节），验证降级行为

---

### M-7: worktree 保留策略中"失败/重试 → 保留"可能导致磁盘耗尽

**问题描述：** 设计要求失败/重试时保留 worktree。如果一个任务反复失败重试（如 gate 失败重试 3 次），每次都创建新 worktree 而非复用上一次的，会积累大量 worktree。即使复用，长时间运行后失败任务积累的 worktree 也会占用可观的磁盘空间（每个 worktree 包含完整 node_modules）。

**影响分析：** 磁盘空间耗尽导致编排器和其他系统进程失败。

**建议修改：**
1. 明确：重试时复用同一个 worktree，不创建新的
2. 增加 worktree 总数上限配置（如 `max_worktrees: 10`），超过时清理最旧的失败 worktree
3. 增加 worktree 保留 TTL（如 24 小时后自动清理）
4. 在 stats.sh 或 dashboard 中显示 worktree 磁盘占用

---

## 🟢 建议（提升质量和可维护性）

### L-1: Bash 脚本不写单元测试的决定值得商榷

**问题描述：** 设计明确说"Bash 脚本不写单元测试，通过手动验证覆盖"。但 dashboard.sh、dev-loop.sh 等包含重要的逻辑（worktree 管理、进程管理、信号处理）。手动验证不可复现、易遗漏。

**建议修改：**
1. 核心 Bash 逻辑（ensure_worktree、cleanup_worktree、进程 kill 逻辑）值得写集成测试
2. 可用 [bats](https://github.com/bats-core/bats-core) 框架，或在 Python 测试中通过 `subprocess` 调用验证
3. 至少为 dev-loop.sh 的关键函数写 smoke test：创建 worktree、清理 worktree、hook 执行

---

### L-2: detect.py 的 CI 配置解析过于脆弱

**问题描述：** `_parse_github_actions` 从 CI YAML 中按关键词 `test/lint/build/check/tsc` 提取命令。这个方法会误提取很多无关命令（如 `actions/checkout`、`upload-artifact`），也可能遗漏使用 Makefile target 包装的实际命令。

**建议修改：**
1. 只提取 `run:` 字段中的命令，且过滤掉明显的 setup 命令（`actions/*`、`npm ci`、`pip install`）
2. 优先提取名称含 "test"/"lint"/"check" 的 step
3. 增加测试用例：用 5-10 个真实的 CI 配置文件验证提取准确率

---

### L-3: 通知模块缺少去重/限流

**问题描述：** 如果一个任务快速连续失败 3 次，会发送 3 条 warn 通知 + 1 条 critical 通知。飞书 webhook 可能触发限流，操作者也会被通知轰炸。

**建议修改：**
1. 增加最小通知间隔配置（如同一任务 5 分钟内只通知一次）
2. 合并通知：3 次失败合并为一条 "任务 #57 连续失败 3 次"
3. 飞书 webhook 增加限流保护（如每分钟最多 5 条）

---

### L-4: 测试策略缺少集成测试和端到端测试

**问题描述：** 测试策略只提到 unittest 覆盖各个 Python 模块。但 openNiuMa 的核心价值在于模块间的协作——state + reconcile + failure + retry + notify 的联动。纯单元测试无法覆盖这些交互路径。

**建议修改：**
1. 增加集成测试层：模拟一个完整的 tick（扫描 inbox → 分配 worker → 检测失败 → 分类 → 重试 → 通知）
2. 增加端到端 smoke test：init.sh → 放入一个简单任务 → dev-loop.sh 执行一轮 → 验证 stats.json 有记录
3. 这些测试可以用 Python subprocess 驱动 bash 脚本来实现

---

### L-5: session log 50 行的硬编码值应可配置

**问题描述：** failure.py 分析 session log 尾部 50 行。这个值对于简短的 lint 错误可能足够，但对于 build 错误（可能有大量 TypeScript 类型错误输出）可能不够。

**建议修改：**
1. 将 50 行改为 `workflow.yaml` 中的 `failure.log_tail_lines` 配置项（默认 50）
2. 或者使用更智能的策略：从尾部向上搜索，直到找到 "exit code" 或 "error" 标记

---

### L-6: 缺少对 Python 3.9 最低版本的 CI 验证

**问题描述：** 依赖声明 Python 3.9+，但没有在测试中验证。如果开发者使用 3.12 的语法特性（如 `match/case`、`type` 关键字），在 3.9 上会失败。

**建议修改：**
1. 在 CI 中增加 Python 3.9 环境的测试矩阵
2. 或在代码规范中明确禁止使用 3.10+ 的语法特性

---

## 总体评价

### 优点
1. **架构设计合理：** 将 19 个内联 Python 块提取为独立模块，是可测试性的质的飞跃
2. **失败分类思路好：** 6 类失败分类 + 差异化策略，比统一重试高级得多
3. **可移植性设计完整：** init.sh 三层策略 + detect.py 自动探测，降低了上手门槛
4. **关注点分离清晰：** 引擎 vs 项目配置的分离原则贯穿始终

### 风险
1. **并发安全是最大隐患：** 多 worker 场景下的 state 文件锁、stats 并发写入、kill 竞态窗口——这些都是必须在 Phase 2 之前彻底解决的问题
2. **关键词匹配的失败分类可靠性存疑：** 建议尽早收集真实 session log 数据来校准关键词和阈值
3. **测试策略偏轻：** 纯单元测试对编排系统来说覆盖不足，至少需要模块间集成测试

### 建议实施优先级

| 优先级 | 编号 | 说明 |
|--------|------|------|
| P0（阻塞实现） | S-1, S-2 | 并发安全问题，不解决则多 worker 模式不可用 |
| P0（阻塞实现） | S-4 | 暂停 loop 的恢复路径，不解决则无人值守场景不可用 |
| P1（Phase 2 前解决） | S-3, M-1, M-3, M-4 | 失败分类准确性和数据完整性 |
| P2（Phase 3 前解决） | M-2, M-5, M-6, M-7 | 配置校验、取消交付、AI 生成校验、磁盘管理 |
| P3（持续改进） | L-1 ~ L-6 | 测试补充、去重限流、可配置化 |

**总评：设计方案的架构方向正确，功能覆盖全面。主要短板在并发安全和测试策略两个方面——这对一个多 worker 编排系统来说恰恰是最核心的质量属性。建议在 Phase 1-2 的实现中把 S-1/S-2/S-4 作为第一优先级解决，并在 Phase 2 完成后补充集成测试，再进入 Phase 3。**
