# spec: 多 Worker 并行调度

## 概述

`dev-loop.sh` 支持多 worker 并行执行任务队列。每个 worker 是一个独立子进程，拥有独立的 git worktree、PostgreSQL 数据库和 state 文件，互不干扰。调度器负责任务认领、worker 生命周期管理和结果同步。

## 架构

```
┌───────────────────────────────────────────────┐
│              parallel_main()                  │
│               并行调度器                       │
│                                               │
│  ┌─── 主循环 (while true) ────────────────┐   │
│  │ 1. process_inbox()  — 检查停止信号     │   │
│  │ 2. 回收已完成 worker                   │   │
│  │ 3. refresh_backlog   — 刷新待办列表    │   │
│  │ 4. 重置孤儿任务     — in_progress→pending │ │
│  │ 5. find_ready_task_ids → claim → 启动  │   │
│  │ 6. sleep 等待                          │   │
│  └────────────────────────────────────────┘   │
│                    │                          │
│      ┌─────────────┼─────────────┐            │
│      ▼             ▼             ▼            │
│  Worker #1     Worker #2     Worker #N        │
│  (子进程)      (子进程)      (子进程)         │
└───────────────────────────────────────────────┘
```

每个 Worker 子进程：

```
$0 --single-task <task_id>
  │
  ├── create_worker_state()   → loop/workers/<id>/state.json
  ├── ensure_worktree()       → .trees/loop-<slug>/
  │     ├── git worktree add --detach
  │     ├── cp -Rc node_modules（APFS clone）或 npm install
  │     ├── createdb poi_dev_loop_<slug>
  │     └── 配置 backend/.env
  └── 进入主循环（串行模式，phase 状态机）
        DESIGN_IMPLEMENT → VERIFY → FIX → MERGE → MERGE_FIX → Done
```

## 入口与模式选择

```bash
# 启动方式
./loop/dev-loop.sh                    # 默认 5 worker 并行
./loop/dev-loop.sh --workers=3        # 指定 3 worker
./loop/dev-loop.sh --workers=1        # 降级为串行模式
```

入口路由逻辑（第 1352–1365 行）：

| 条件 | 模式 |
|---|---|
| `MAX_WORKERS > 1` 且无 `--single-task` | → `parallel_main()`（调度器） |
| `--single-task <id>` | → 单任务 worker（独立 state） |
| `MAX_WORKERS = 1` | → 串行主循环 |

## 核心机制

### 1. 状态锁（并发安全）

多 worker 共享一份主 `loop-state.json`，通过 `mkdir` 原子锁保证并发安全：

```bash
lock_state()      # mkdir loop/.locks/state（自旋重试，10s 超时强制清除）
unlock_state()    # rmdir loop/.locks/state
locked_state_op() # lock → python3 修改 state → unlock
```

macOS 无 `flock` 命令，`mkdir` 是 POSIX 原子操作，跨平台兼容。

### 2. 任务认领（原子操作）

```
find_ready_task_ids()  →  遍历 queue，筛选 status=pending 且依赖已满足的任务
claim_task(id)         →  加锁 → 将 status 改为 in_progress → 解锁
```

同一时刻只有一个 worker 能成功认领同一个任务。

### 3. Worker State 隔离

`create_worker_state()` 从主 state 提取单个任务信息，写入 `loop/workers/<id>/state.json`：

```json
{
  "dev_branch": "dev/xxx",
  "current_item_id": 3,
  "current_phase": "DESIGN_IMPLEMENT",
  "queue": [{ /* 仅当前任务 */ }],
  "completed": [/* 全局已完成列表（用于依赖判断）*/],
  "worktree_path": null,
  "main_repo_path": "/path/to/repo"
}
```

Worker 子进程将 `STATE_FILE` 指向自己的 state 文件，后续读写完全独立。

### 4. Worktree 隔离

每个 worker 的代码操作在独立 git worktree 中进行：

| 项目 | 路径 |
|---|---|
| Worktree 目录 | `.trees/loop-<slug>/` |
| 数据库 | `poi_dev_loop_<slug>` |
| 环境变量 | `.trees/loop-<slug>/backend/.env` |

worktree 创建优化：
- **APFS clone**：`package-lock.json` 一致时用 `cp -Rc`（macOS CoW，秒级完成）
- **回退**：lock 不一致才跑 `npm install --prefer-offline`
- **复用**：目录已存在时直接复用，不重建

### 5. 结果同步

Worker 完成后，调度器执行：

```
mark_task_done(id, worker_state)   → 更新任务 markdown 文件（frontmatter + 完成总结）
sync_worker_result(id, exit_code)  → 加锁修改主 state：
  - exit=0 且 phase 完成 → status=done，移入 completed
  - 否则 → status=pending（允许重试）
  - 传播 BLOCKED 状态给下游依赖任务
cleanup_worktree(slug)             → 删除 worktree + dropdb
```

### 6. 孤儿任务恢复

每轮循环检测 `in_progress` 但无对应活跃 worker 的任务，自动重置为 `pending`：

```
场景：worker 异常退出（OOM / kill -9），PID 文件已被清理但 state 中仍为 in_progress
恢复：下一轮循环自动检测并重置，后续 worker 可重新认领
```

## 信号处理与优雅退出

```bash
trap cleanup_all_workers INT TERM HUP
```

收到终止信号时：
1. 遍历 `workers/*/pid`，`kill` 每个 worker 进程
2. `pkill -P` 杀掉 worker 的 claude 子进程
3. `wait` 等待全部退出
4. 调度器 `exit 0`

## 调度策略

```
┌─── 每轮循环 ────────────────────────────────────────┐
│                                                      │
│  有活跃 worker？                                     │
│    ├── 是 → sleep 10s → 下一轮                       │
│    └── 否 → 检查剩余任务：                           │
│           ├── 无 pending → "所有任务完成！" → break   │
│           ├── 全部等依赖 → sleep 60s → 下一轮        │
│           └── 有可认领的 → sleep 2s → 下一轮         │
│                                                      │
│  认领上限：active_count < MAX_WORKERS                │
│  跳过条件：已有 pid 文件 / claim_task 返回非 ok      │
└──────────────────────────────────────────────────────┘
```

## 文件结构

```
loop/
├── dev-loop.sh           # 主脚本（调度器 + worker 逻辑）
├── loop-state.json       # 主状态文件（队列、已完成、已阻塞）
├── workers/              # Worker 运行时目录（启动时清空）
│   ├── <task_id>/
│   │   ├── state.json    # Worker 独立 state
│   │   ├── pid           # Worker 进程 PID
│   │   └── name          # 任务名（用于日志）
├── logs/
│   └── worker-<id>.log   # 每个 worker 的输出日志
├── .locks/
│   └── state/            # mkdir 互斥锁（目录存在 = 已加锁）
└── tasks/                # 任务描述文件（markdown）

.trees/                   # Worktree 父目录（在主仓库外）
├── loop-<slug-a>/        # Worker A 的隔离代码目录
├── loop-<slug-b>/        # Worker B 的隔离代码目录
└── ...
```

## 限制与注意事项

1. **锁粒度**：全局一把锁（`lock_state`），高并发时可能成为瓶颈，但当前 MAX_WORKERS=5 足够
2. **磁盘开销**：每个 worktree 含完整 node_modules（APFS clone 不占额外空间，仅限 macOS）
3. **数据库资源**：每个 worker 创建独立 PG 数据库，完成后自动 `dropdb` 清理
4. **日志分散**：每个 worker 日志独立文件，需按 task_id 查找对应日志
5. **不支持跨 worker 依赖的实时通信**：依赖通过主 state 的 `completed` 列表判断，轮询间隔决定感知延迟
